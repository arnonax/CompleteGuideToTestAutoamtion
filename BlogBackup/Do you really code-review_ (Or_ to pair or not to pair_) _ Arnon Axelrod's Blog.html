Do you really code-review? (Or: to pair or not to pair?)

February 10, 2012

facebook linkedin twitter email
tags: Code review, DEV, Pair programming
no comments

There’s no doubt that Code review is an important practice. It has many advantages:

· Bugs are likely to be found early

· The quality and especially the readability of the code are kept high

· Technical knowledge and experience is shared

· Domain knowledge is shared

In most organizations I encountered, some kind of code-review policy was in place. However, not many of them succeeded in implementing the policy. The main “excuse” is usually pressure, but in my opinion, the problem is deeper than this as I’ll explain later.

Another practice, which is closely related to code review, but is much less popular, is pair-programming (where 2 programmers sit together near one computer and walk through the task together). Most managers I talked to reject this practice outright, claiming that it’s very wasteful and not at all cost effective.

In this article I’ll try to compare the 2 practices, showing that pair-programming is often preferable than regular code-review policy.

Online code-review vs. offline code-review

Generally speaking, there are main 2 ways to perform a code review. Each of which has its own advantages and drawbacks.

The first way is often called "online code-review" (a.k.a. face-to-face code review). Online code review is when the reviewer and reviewee look at the code together and discuss it (usually by sitting near the same computer, looking at the same screen).

The second way is the opposite, and called "offline code review". Offline code review is when the reviewee sends the changes to the reviewer (either through mail, shelve-set or using special tools), which then reviews it in his own time, and send his comments back to the reviewee.

Sometimes the policy dictates which way to use, and sometimes it is left for the developers to decide and even combine the 2 ways in any way that fits best.

Why code-review doesn’t work?

Well… it can work, and sometimes it is, but either it’s done sloppily, or it takes much more time than you might think. In addition, there are other aspects that play a negative part in code-reviews: ego and politics.

Let’s first examine the objective reasons that impede a good code review policy. In order to examine this, let’s consider a scenario where the code review is done perfectly, and assume that the developers can choose which style of code review to use. We’ll pay attention especially to the time each step takes, both to the reviewee and the reviewer.

I never asked this question, but I guess that if I would ask managers (and even developers) how much % of the development time should be spent by code review, the answer would be around 10%. How much would you say? Let’s examine it together!

Stage 1: preparation

Before the code review itself, when the developer knows that he has to prepare his code for code review he invests all his efforts to come prepared to the review. For that, he’s self-reviewing his code, applying refactoring and ensuring his code is as beautiful as he can make it (which is great!). He also tests his code and applies any check-in policy measures to ensure that everything is ok.

Often, after making sure that his changes are state-of-the-art, he needs to get the latest changes from the source-control repository and merge his code with them. If there were conflicts, the developer has to go over again and make sure that everything still works.

Stage 2: Calling for a review

Once the reviewee is ready, he needs to ask the reviewer to review his code. Usually, the reviewer, which is another developer, will be in the middle of something and will only be available after some time. During this time, the developer (the reviewee) can’t do any work that is related or based on the task that is waiting for review. In other words, the developer simply has to sit and wait (maybe read emails etc…)

Stage 3: Explaining the problem and the intended solution

When the reviewer is finally ready and comes over, the first thing that must be done before diving into the code is to explain to the reviewer the problem that the reviewee tried to solve and how, in high-level, he solved it. Note that if the reviewer disagree with the solution (and convinces the reviewee that he’s right), then all of the work of the reviewee is wasted, and he has to start all over again!

This discussion is best done "online" (face-to-face) in order to make sure that the reviewer and reviewee understand each other. This means that the time this discussion takes is X2 developers.

After the reviewer understands the high-level solution, there are two options: either to continue the review face-to-face, or to go "offline".

Stage 4.1: Online code review

If the reviewer and reviewee continue to do the review online, then the reviewer should drive the review, in order to ensure that the code is readable and self-explanatory. While this approach takes the time of both the reviewer and reviewee, it allows them to perform small changes on-the-fly. A good review should be thorough and examine each line that was changed or added. Of course that this kind of code-review takes the time of both developers.

More often than not, the reviewer provide comments that require some changes which takes too long to be done during the review, and these changes are kept for the next round. These changes will require an additional review after they’re done.

Stage 4.2: Offline code review

The reviewer and reviewee may choose to continue the review offline instead of online. In this case the reviewer reads the code (or the changes) himself, and writes down his comments. After he’s done, he sends the comments back to the reviewee. Note that even though that the reviewee can do other things during this time, he can’t really do any progress on tasks that are related or rely on the reviewed code, and so he has to wait…

Stage 5: Fixing the comments

Now the developer has to fix all the issues that the reviewer raised. After doing so, he needs to test everything; then get latest sources from source control, merge, and test again once more. Then he needs to call the reviewer again and wait for him to become available…

Stage 6: reviewing the fixes and check-in

After the developer fixed the issues that the reviewer has raised, the reviewer should review these fixes too, to see that they’re ok.

Given that this review yields no major comments, the developer should now get latest sources for the last time, merge, test, and finally check-in. (given also that the merge was straight-forward).

Small chucks or big chunks?

One may claim that if changes that are to be reviewed each time are small (i.e. split the task into more fine-grained sub tasks) then the amount of fixes and wasted time is decreased. I tend to agree, but on the other hand it makes context-switches of the reviewer more often, and the total amount time spent on code-review is not much shorter in total, as this loop happens more often.

Subjective drawbacks of code review

In addition to the overhead that a good code-review procedure implies, code reviews also suffer from 2 major flaws that are related to the human nature

Ego

Often, the comments raised in a code review are those related to style or personal preferences, and not to essential stuff. Sometimes, often unconsciously, people treat code-reviews as a competition for who’s a better programmer, or who’s right and who’s wrong. Because the reviewee, which is the developer that implemented the task is usually considered the owner of that task, the reviewer doesn’t share the sense of responsibility for the success of delivering the task on time and on quality.

It’s often the case that because of time spent on the less important matters, that less time is spent on the more important stuff which is then reviewed sloppily.

Politics

Naturally, people have peers that they consider as friends, and peers that they’re not getting along very well with them. Either consciously or unconsciously, reviewers tend to be more “forgiving” to their friends, and too relentless to their foes. None of which yields a good, professional review.

Benefits of pair-programming

Let me clarify this first: Pair-programming shouldn’t be forced. There are people that are not feeling comfortable in this situation, or don’t get along with someone else. However, IMHO, it should be encouraged, and also it should be encouraged that people should pair each time with someone else (this is a good practice also for code reviews).

In addition to the benefits of code-review, pair-programming has these advantages:

· Two are better than one in solving problems. When a single person is faced with a problem, he’ll likely to go with the first solution that pops in his mind, not considering alternatives. When two people are faced with a problem, usually each of them thinks of a different solution. This causes them to discuss the pros and cons of each way and to choose which one is better. Often, the result they’ll choose is neither of their initial ideas, but rather a combination of them with the most advantages and least disadvantages.

· Two people sitting together to solve a problem are much more focused at solving that problem and are much less likely to be distracted or to solve the wrong problem.

· Because a code review is done after the fact, any comment feels like a criticism (with nobody likes). Pair-programming on the other hand, feels more like teaching and learning, which people are often much more open for.

· The sense of responsibility for the task is shared between the 2 peers, so they’re more likely to co-operate instead of compete. In addition, this shared sense of responsibility improves their interpersonal relationships, resulting in a more integrated team.

· While code-review aids in knowledge transfer, it is done only on the end product of the task. Knowledge and practices about things like way of thinking, usage of tools etc. are not shared. In pair-programming, the knowledge and experience is shared throughout the way. This helps tremendously improving the skills of the younger developers and raises the level of the entire team in a very short period of time!

Conclusion

In spite of all I wrote above, I think that code review is a very important and beneficial practice. However, as I showed here, it “costs” much more than most managers realized. Because of that, they don’t really encourage a really good implementation of the policy and so the policy is not really enforced. If you consider the real cost, you may come to find that the difference between the cost of a good implementation of a code-review policy and the cost of pair-programming is not that different. For example, if a task takes the developer 4 days to implement, and then 2 more days to complete the code review process, during which the reviewer is occupied with the code review for 1 day too, then in total this task has taken 7 man days, whilst if the 2 programmers would have working on the task together from the start, then they might finish it in 3 days (which are 6 man days).

So as I see it, after reading this post so far, as a manager you have 4 options:

  1. Pretend you haven’t read this post, and live in lie
  2. Choose to keep a sloppy code review process (or not at all), but admit it. This will be the cheapest solution in the short term, but you’ll miss most of the benefits of the practice, which usually pay off in the long term. If this is what you choose, let me ask you to be honest and open about it, and not try to pretend that you’re maintaining a strict policy while blaming your developers for not adhering to it.
  3. Choose to be strict about the code review process, and be willing to let your developers the time necessary to adhere to the policy. It will be costly, but in the long term you’ll gain the important mentioned benefits
  4. Encourage pair-programming as a better alternative to code reviews. It will cost more-or-less the same as being strict about the code review, but it will sure pay in the long run, due to the additional benefits mentioned above.

So which one do you choose…?
